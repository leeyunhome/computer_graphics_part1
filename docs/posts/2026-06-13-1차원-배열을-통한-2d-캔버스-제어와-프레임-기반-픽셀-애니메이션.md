# 1차원 배열을 통한 2D 캔버스 제어와 프레임 기반 픽셀 애니메이션

학습자가 작성한 이번 커밋은 그래픽스 파이프라인의 기초인 **프레임 버퍼(Frame Buffer)**를 CPU 측에서 직접 제어하고, 매 프레임 변화를 주어 동적인 화면(애니메이션)을 만들어내는 핵심 개념을 담고 있습니다.

---

### 1. 학습자가 직접 구현하며 이해한 개념 분석

학습자는 단순히 정적인 이미지를 화면에 그리는 단계를 넘어, 시간의 흐름(프레임 업데이트)에 따라 화면의 데이터가 어떻게 변화하는지 직접 실험하고 이해했습니다.

*   **2D 이미지의 1차원 메모리 매핑**: 컴퓨터 메모리는 선형(1차원) 구조입니다. 학습자는 `canvasWidth * canvasHeight` 크기의 1차원 `std::vector` 공간이 2D 화면과 어떻게 대응하는지 이해하고 있습니다.
*   **프레임별 상태 보존 (State Retention)**: `static int i` 변수를 활용하여, `Update()` 함수가 매 프레임 호출될 때마다 초기화되지 않고 이전 프레임의 픽셀 위치 상태를 유지하도록 구현했습니다.
*   **더블 버퍼링과 화면 초기화**: 매 프레임마다 `pixels` 벡터를 회색(`Vec4{0.8f, 0.8f, 0.8f, 1.0f}`)으로 새로 채워줌으로써, 이전 프레임의 잔상이 남지 않고 '빨간 점 하나만 이동하는' 깔끔한 애니메이션 효과를 구현했습니다.
*   **경계 조건 처리 (Boundary Handling)**: 배열의 인덱스 범위를 초과하여 프로그램이 크래시(Crash)나는 것을 방지하기 위해, 인덱스가 벡터의 끝(`pixels.size() - 1`)에 도달하면 다시 `0`으로 순환(Wrap-around)시키는 예외 처리를 올바르게 수행했습니다.

---

### 2. 핵심 수식 및 개념

컴퓨터 그래픽스에서 2D 좌표 $(x, y)$와 메모리 상의 1차원 인덱스 $i$ 사이의 변환은 가장 기본적이면서도 중요한 수식입니다. 학습자는 코드 내에서 이 변환 관계를 은연중에 활용하고 있으며, 최종적으로 1차원 인덱스 $i$를 가로축 $x$와 세로축 $y$로 분해하여 생각할 수 있습니다.

$$ i = x + (y \times \text{canvasWidth}} $$

반대로, 1차원 인덱스 $i$로부터 현재 픽셀의 2D 스크린 좌표 $(x, y)$를 도출하는 역변환 수식은 다음과 같습니다.

$$ x = i \pmod{\text{canvasWidth}} $$
$$ y = \lfloor \frac{i}{\text{canvasWidth}} \rfloor $$

이 수식을 통해 학습자가 구현한 빨간 점은 단순히 오른쪽으로 가다가 가로 끝에 도달하면 다음 줄의 첫 번째 칸(우측 끝에서 좌측 끝 다음 줄)으로 넘어가는 **래스터 스캔(Raster Scan)** 순서대로 움직이게 됩니다.

---

### 3. 알고리즘 흐름 (의사코드)

학습자가 작성한 프레임 업데이트 로직을 추상화된 흐름으로 나타내면 다음과 같습니다.

```text
function UpdateFrame():
    // 1. 프레임 버퍼를 기본 배경색(회색)으로 초기화
    FillFrameBuffer(color=GRAY)

    // 2. 현재 애니메이션 진행 상태(인덱스 i)에 해당하는 위치에 빨간색 주입
    SetPixelColor(index=i, color=RED)

    // 3. 인덱스 업데이트 및 경계 검사 (환형 구조)
    if i이 (최대 픽셀 수 - 1)과 같다면:
        i = 0
    else:
        i = i + 1

    // 4. 완성된 CPU 프레임 버퍼를 GPU 텍스처 버퍼로 업로드
    UploadToGPUTexture()
```

### 4. WebGPU 인터랙티브 데모

만약 이 매커니즘을 브라우저 표준 그래픽 API인 **WebGPU**의 **Compute Shader**를 활용해 구현한다면 어떤 모습일까요?

#### 데모 시각화 모습
*   **화면 구성**: 브라우저 화면에 $320 \times 240$ 등의 작은 캔버스가 렌더링됩니다. 배경은 연한 회색으로 가득 차 있습니다.
*   **애니메이션**: 왼쪽 상단 $x=0, y=0$ 위치에서 빨간색 점 하나가 시작하여 오른쪽으로 빠르게 이동합니다. 우측 끝에 도달하면 즉시 한 줄 아래 왼쪽 끝으로 내려가 이동을 계속하며, 화면 오른쪽 아래 끝에 도달하면 다시 처음 위치로 돌아와 무한히 반복됩니다.
*   **CPU-GPU 역할 분담의 변화**: 
    학습자의 DX11 코드에서는 **CPU**가 루프를 돌며 픽셀 값을 쓰고 GPU로 매 프레임 복사(Copy)하는 방식을 썼지만, WebGPU 데모에서는 GPU의 **Compute Shader**가 유니폼 버퍼(Uniform Buffer)로부터 현재 프레임 번호(`frame_count`)를 받아 병렬로 캔버스에 직접 픽셀 값을 쓰게 됩니다.

#### WebGPU Compute Shader 핵심 구조 (WGSL 예시)
```rust
@group(0) @binding(0) var<uniform> frame_count : u32;
@group(0) @binding(1) var out_texture : texture_storage_2d<rgba8unorm, write>;

@compute @workgroup_size(16, 16)
fn main(@builtin(global_invocation_id) global_id : vec3<u32>) {
    let size = textureDimensions(out_texture);
    let total_pixels = size.x * size.y;
    
    // 1차원 타겟 인덱스 계산 (프레임 수에 따라 순환)
    let target_index = frame_count % total_pixels;
    
    // 현재 GPU 쓰레드가 담당하는 픽셀의 1차원 인덱스 계산
    let current_index = global_id.x + (global_id.y * size.x);
    
    var color = vec4<f32>(0.8, 0.8, 0.8, 1.0); // 기본 배경 회색
    
    if (current_index == target_index) {
        color = vec4<f32>(1.0, 0.0, 0.0, 1.0); // 빨간색 픽셀 애니메이션
    }
    
    textureStore(out_texture, global_id.xy, color);
}
```

WebGPU로 이 데모를 구동하면 CPU 대역폭을 소모하는 버퍼 복사 연산 없이, GPU 내부에서 모든 픽셀 계산과 렌더링이 초당 60프레임 이상으로 가볍고 부드럽게 돌아가는 웹 그래픽스의 강력함을 직접 확인할 수 있습니다.
