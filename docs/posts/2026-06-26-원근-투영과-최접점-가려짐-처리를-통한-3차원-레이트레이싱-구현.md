# 원근 투영과 최접점 가려짐 처리를 통한 3차원 레이트레이싱 구현

이번 커밋은 정투영(Orthographic Projection)에서 **원근 투영(Perspective Projection)**으로 전환하고, 여러 오브젝트가 레이 상에 겹쳐 있을 때 가장 가까운 오브젝트만 화면에 그리는 **최접점 판정(Occlusion Sorting/Depth Testing)** 알고리즘을 도입한 중요한 단계입니다. 

컴퓨터 그래픽스 전문가의 시각에서 이 커밋의 핵심 개념과 수학적 배경, 그리고 코드 상의 중요한 개선점을 심층 분석합니다.

---

## 1. 핵심 개념 및 수학적 원리

### 1) 원근 투영 (Perspective Projection)과 광선 생성
기존의 정투영 방식은 스크린의 모든 픽셀에서 동일한 방향 $$(0, 0, 1)$$로 평행한 광선을 투사했습니다. 반면, 원근 투영은 하나의 관찰 위치(눈/카메라)인 $$\mathbf{E}$$ (여기서는 `eyePos = (0, 0, -1.5)`)로부터 스크린 상의 픽셀 위치 $$\mathbf{P}_w$$를 향해 사방으로 퍼져나가는 광선을 생성합니다.

수식으로 표현하면, 카메라 위치 $$\mathbf{E}$$와 월드 공간의 픽셀 위치 $$\mathbf{P}_w$$가 주어졌을 때, 픽셀을 통과하는 광선의 방향 벡터 $$\mathbf{d}$$는 다음과 같이 정의됩니다.

$$ \mathbf{d} = \frac{\mathbf{P}_w - \mathbf{E}}{\|\mathbf{P}_w - \mathbf{E}\|} $$

학습자는 스크린 평면(z=0) 상의 점인 `pixelPosWorld`를 광선의 시작점(Origin)으로 설정하고, 위에서 계산한 방향 벡터 `rayDir`을 부여하여 광선 $$\mathbf{R}(t)$$를 정의했습니다.

$$ \mathbf{R}(t) = \mathbf{P}_w + t\mathbf{d} \quad (t \ge 0) $$

*(참고: 카메라 위치 $$\mathbf{E}$$를 광선의 시작점으로 잡고 $$\mathbf{P}_w$$ 방향으로 투사해도 수학적으로 동일한 방향을 얻을 수 있지만, 스크린 이전의 불필요한 연산을 줄이기 위해 스크린 상의 점을 시작점으로 잡는 기법을 사용했습니다.)*

### 2) 가려짐 처리 (Occlusion Sorting)
3차원 공간에서 광선이 진행할 때 여러 개의 물체와 충돌할 수 있습니다. 화면에 올바른 깊이감을 표현하려면 카메라와 가장 가까운 위치에서 발생한 충돌(최소 양수값 $$t$$)을 선택해야 합니다.

$$ t_{\text{closest}} = \min \{ t_i \mid t_i \ge 0 \} $$

## 2. 코드 구조 분석 및 구현의 의의

기존 렌더러는 레이가 구체 하나에 부딪히자마자 즉시 탐색을 종료하고 색상을 반환했습니다. 그러나 이번 변경을 통해 장면 내 모든 오브젝트(`objects`)를 순회하며 충돌 거리를 비교 분석하는 구조로 확장되었습니다.

### 퐁 반사 모델 (Phong Reflection Model)의 완성도 유지
충돌점에서의 색상을 계산할 때는 정교한 **퐁 반사 모델(Phong Reflection Model)**이 적용되었습니다. 
* **Ambient (환경광)**: 빛이 직접 닿지 않아도 존재하는 기본적인 밝기.
* **Diffuse (난반사광)**: 표면의 법선 벡터 $$\mathbf{N}$$과 광원 방향 벡터 $$\mathbf{L}$$의 내적을 이용해 빛을 받는 각도에 따른 밝기 표현 ($$\mathbf{N} \cdot \mathbf{L}$$).
* **Specular (경면반사광)**: 반사 벡터 $$\mathbf{R}$$과 시선 반대 방향 벡터 $$\mathbf{V}$$의 내적 및 신코사인 지수($$\alpha$$)를 활용한 하이라이트 표현 ($$(\mathbf{R} \cdot \mathbf{V})^\alpha$$).

이 일련의 조명 공식은 `traceRay` 함수 내에서 가장 가까운 충돌 정보(`Hit`)를 바탕으로 정확하게 계산됩니다.

## 3. 코드 코멘트 및 치명적인 버그(Bug) 분석

학습자의 코드에서 매우 흥미롭고 교육적인 **논리적 오류(Bug)**가 발견되었습니다. 학습자는 최접점을 구하기 위해 `closestD` 변수를 선언했으나, 정작 루프 내에서 **새로 구한 충돌 거리가 기존 최단 거리보다 짧은지 검사하는 조건식**을 누락했습니다.

### 문제 코드 분석
```cpp
// Raytracer.h - FindClosestCollision 중 일부
float closestD = 1000.0; // inf
Hit closestHit = Hit{ -1.0, dvec3(0.0), dvec3(0.0) };

for (int l = 0; l < objects.size(); l++)
{
    auto hit = objects[l]->CheckRayCollision(ray);

    if (hit.d >= 0.0f) // <- 버그 발생 지점! hit.d < closestD 조건이 빠져 있음
    {
        closestD = hit.d;
        closestHit = hit;
        closestHit.obj = objects[l];
    }
}
```

### 왜 이 버그가 발생했는가?
이 조건문은 단순히 충돌 여부(`hit.d >= 0.0f`)만 판단하고 있습니다. 즉, 루프를 돌면서 충돌하는 물체가 발견될 때마다 `closestD`를 그냥 덮어씌워 버립니다. 결과적으로 **가장 가까운 오브젝트가 아니라, 리스트에서 가장 마지막에 위치하며 충돌이 감지된 오브젝트**가 최종 선택됩니다.

### 그럼에도 결과(result.png)가 정상처럼 보였던 이유 (우연의 일치)
생성자에서 오브젝트를 추가한 순서를 보면 이 의문이 풀립니다.
1. `sphere3` (Blue, $z = 1.5$)
2. `sphere2` (Green, $z = 1.0$)
3. `sphere1` (Red, $z = 0.5$) - 가장 가까움

루프는 index 0(`sphere3`)부터 2(`sphere1`)까지 순서대로 돕니다.
* $l=0$ 일 때: `sphere3`과 충돌 감지 $\rightarrow$ `closestHit`은 `sphere3`이 됨.
* $l=1$ 일 때: `sphere2`와 충돌 감지 $\rightarrow$ `closestHit`은 `sphere2`로 덮어씌워짐.
* $l=2$ 일 때: `sphere1`과 충돌 감지 $\rightarrow$ `closestHit`은 `sphere1`로 최종 덮어씌워짐.

공교롭게도 **가장 가까운 구체가 리스트의 가장 마지막에 처리되었기 때문에** 화면에는 정상적으로 붉은색 구체가 가장 앞에 그려진 것처럼 보였던 것입니다. 만약 구체의 추가 순서를 바꾸거나 동적으로 움직인다면 렌더링이 깨지게 됩니다.

## 4. 올바른 최접점 알고리즘 (의사코드)

이를 올바르게 수정하려면 충돌 거리인 `hit.d`가 양수이면서 동시에 **현재까지 저장된 `closestD`보다 작을 때만** 값을 갱신해야 합니다.

```cpp
// 올바른 최접점 탐색 의사코드 (Pseudo-code)
function FindClosestCollision(ray) {
    closestD = infinity
    closestHit = empty_hit

    for each object in objects {
        hit = object.CheckRayCollision(ray)
        
        // 0보다 크고(전방 충돌), 현재 저장된 최단 거리보다 짧은 경우에만 갱신
        if (hit.distance >= 0.0 and hit.distance < closestD) {
            closestD = hit.distance
            closestHit = hit
            closestHit.object = object
        }
    }
    return closestHit
}
```

## 5. WebGPU 인터랙티브 데모로의 확장 시각화

이 원근 투영 레이트레이싱 알고리즘을 최신 웹 그래픽스 표준인 **WebGPU** 기반의 **Compute Shader**로 마이그레이션하면, CPU 병렬 처리(OpenMP)를 압도하는 실시간 60fps 인터랙티브 렌더링을 구현할 수 있습니다.

### 1) 구조 설계
* **Storage Buffer**: 구체들의 데이터(위치, 반지름, 퐁 반사 계수 등)를 GPU 버퍼에 업로드합니다.
* **Storage Texture**: 계산된 픽셀 색상을 직접 쓸 수 있는 쓰기 가능 텍스처(Write-only texture)를 바인딩합니다.
* **Compute Shader (WGSL)**: CPU에서 수행하던 `Render()`의 이중 루프를 GPU의 수천 개 스레드로 분할 처리합니다.

### 2) WGSL 셰이더 핵심 파이프라인 예시
```wgsl
struct Sphere {
    pos: vec3<f32>,
    radius: f32,
    ambient: vec3<f32>,
    diffuse: vec3<f32>,
    specular: vec3<f32>,
    alpha: f32,
};

@group(0) @binding(0) var<storage, read> spheres: array<Sphere>;
@group(0) @binding(1) var outputTex: texture_storage_2d<rgba8unorm, write>;

@compute @workgroup_size(16, 16)
fn cs_main(@builtin(global_invocation_id) id: vec3<u32>) {
    let size = textureDimensions(outputTex);
    if (id.x >= size.x || id.y >= size.y) { return; }

    // 1. Screen to World Space 변환 (정규화된 디바이스 좌표계 적용)
    let uv = vec2<f32>(id.xy) / vec2<f32>(size);
    let pixelPosWorld = vec3<f32>((uv.x * 2.0 - 1.0) * aspect, -(uv.y * 2.0 - 1.0), 0.0);
    
    // 2. 원근 투영 광선 생성
    let eyePos = vec3<f32>(0.0, 0.0, -1.5);
    let rayDir = normalize(pixelPosWorld - eyePos);
    
    // 3. 최접점 판정 루프 (버그 수정 버전 적용)
    var closestD = 10000.0;
    var closestHitIndex = -1;
    
    for (var i = 0u; i < arrayLength(&spheres); i++) {
        let d = checkRaySphereCollision(pixelPosWorld, rayDir, spheres[i]);
        if (d >= 0.0 && d < closestD) {
            closestD = d;
            closestHitIndex = i;
        }
    }
    
    // 4. 퐁 반사 모델 조명 계산 및 텍스처 출력
    var color = vec3<f32>(0.0);
    if (closestHitIndex != -1) {
        color = calculatePhongReflection(closestHitIndex, closestD, pixelPosWorld, rayDir);
    }
    
    textureStore(outputTex, id.xy, vec4<f32>(color, 1.0));
}
```

WebGPU 데모에서는 사용자가 웹브라우저 상에서 마우스를 드래그하여 카메라(`eyePos`)의 위치를 실시간으로 바꿀 수 있습니다. 카메라가 움직임에 따라 구체들의 겹침 관계(Occlusion)가 부드럽고 정확하게 실시간으로 변하는 원근감을 직접 체험할 수 있게 됩니다.
