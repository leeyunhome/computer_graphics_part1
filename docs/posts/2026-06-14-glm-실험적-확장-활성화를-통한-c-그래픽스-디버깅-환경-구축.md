# GLM 실험적 확장 활성화를 통한 C++ 그래픽스 디버깅 환경 구축

DirectX 11 및 C++를 이용한 3D 그래픽스 프로그래밍에서 호스트(CPU) 측의 수학 연산 라이브러리로 **GLM(OpenGL Mathematics)**을 사용하는 것은 매우 흔하고 강력한 선택입니다. 이번 커밋은 빌드 오류를 해결하는 단순한 설정을 넘어, 그래픽스 파이프라인에서 복잡한 행렬 및 벡터 연산을 디버깅하기 위한 필수적인 디버깅 도구(`glm/gtx/string_cast.hpp`)를 활성화하는 중요한 의미를 담고 있습니다.

---

## 1. 이 커밋의 핵심 개념: 그래픽스 디버깅과 GLM 확장 기능

3D 그래픽스 애플리케이션을 개발할 때 가장 빈번하게 발생하는 오류는 **좌표계 변환 오차**나 **잘못된 행렬 곱셈 순서**로 인해 화면에 아무것도 그려지지 않는 현상(블랙 스크린)입니다. 

로컬 공간의 정점 $\mathbf{v}_{local}$이 화면(Clip Space)에 최종 투영되는 과정은 다음과 같은 행렬 변환 공식을 따릅니다.

$$\mathbf{v}_{clip} = P \cdot V \cdot M \cdot \mathbf{v}_{local}$$

여기서 각 행렬은 다음과 같습니다.
*   $M$ (Model Matrix): 로컬 좌표계를 월드 좌표계로 변환
*   $V$ (View Matrix): 카메라 기준 좌표계로 변환
*   $P$ (Projection Matrix): 원근감 또는 직교 투영 적용

이 행렬들이 제대로 계산되었는지 확인하려면 16개의 float 값(4x4 행렬)을 콘솔이나 로그에 직접 출력하여 디버깅해야 합니다. GLM은 이를 위해 `glm::to_string()`이라는 매우 편리한 함수를 제공하는데, 이는 **실험적 확장(Experimental Extensions, GTX)** 헤더인 `glm/gtx/string_cast.hpp`에 정의되어 있습니다.

GLM은 의도치 않은 실험적 기능 사용으로 인한 호환성 문제를 방지하기 위해, 사전 정의된 매크로인 `GLM_ENABLE_EXPERIMENTAL`이 선언되어 있을 때만 이 기능들을 노출하도록 설계되어 있습니다. 학습자는 이를 해결하기 위해 프로젝트 전역 설정(`.vcxproj`)의 모든 빌드 구성(Debug/Release, Win32/x64)에 해당 전처리기를 추가하여 빌드 안정성을 확보했습니다.

## 2. 전체 구조에서 이 변경이 갖는 의미

기존에는 빌드 에러로 인해 3D 변환 행렬의 값을 직접 추적하는 데 한계가 있었을 것입니다. 이번 설정을 통해 프로젝트 전역에서 GLM의 다양한 확장 기능을 제약 없이 사용할 수 있게 되었습니다.

특히, DirectX 11은 왼손 좌표계(Left-Handed)를 기본으로 사용하고 행 우선(Row-Major) 방식을 선호하는 반면, GLM은 기본적으로 오른손 좌표계(Right-Handed) 및 열 우선(Column-Major) 방식을 따릅니다. 따라서 두 API를 혼용할 때 전치 행렬(Transpose) 연산이나 좌표계 반전 연산이 필수적입니다.

예를 들어, DirectX 11의 상수 버퍼(Constant Buffer)에 행렬을 업로드하기 전, 다음과 같은 변환이 제대로 이루어졌는지 호스트 측에서 값의 변화를 감시해야 합니다.

$$M_{DX} = (M_{GLM})^{T}$$

이번 변경으로 `glm::to_string()`을 사용할 수 있게 됨으로써, 이러한 변환 과정의 정합성을 CPU 단계에서 완벽하게 검증한 후 GPU로 전송할 수 있는 강력한 무기를 얻게 된 것입니다.

## 3. 핵심 알고리즘 및 활용 추상화 (의사코드)

학습자가 이번 설정을 통해 구현하고자 하는 핵심 디버깅 흐름을 추상화된 의사코드로 표현하면 다음과 같습니다. 3D 정점이 카메라 뷰와 투영을 거쳐 어떻게 변환되는지 콘솔에서 즉시 시각화할 수 있습니다.

```cpp
#include <iostream>
// 전처리기에 GLM_ENABLE_EXPERIMENTAL이 등록되어 있으므로 정상적으로 포함됩니다.
#include <glm/glm.hpp>
#include <glm/gtc/matrix_transform.hpp>
#include <glm/gtx/string_cast.hpp> 

void DebugTransformPipeline()
{
    // 1. 임의의 정점 정의 (로컬 공간)
    glm::vec4 localVertex(0.5f, 0.5f, 0.0f, 1.0f);
    
    // 2. 변환 행렬 생성 (Model, View, Projection)
    glm::mat4 model = glm::translate(glm::mat4(1.0f), glm::vec3(0.0f, 0.0f, 5.0f));
    glm::mat4 view  = glm::lookAt(glm::vec3(0.0f, 0.0f, 0.0f), glm::vec3(0.0f, 0.0f, 1.0f), glm::vec3(0.0f, 1.0f, 0.0f));
    glm::mat4 proj  = glm::perspective(glm::radians(45.0f), 800.0f / 600.0f, 0.1f, 100.0f);
    
    // 3. 합성 변환 행렬 계산 (MVP)
    glm::mat4 mvp = proj * view * model;
    
    // 4. 최종 변환 정점 계산
    glm::vec4 clipVertex = mvp * localVertex;

    // [핵심 디버깅 단계] GLM string_cast를 이용한 실시간 값 추적
    std::cout << "--- Graphics Pipeline Debug Info ---" << std::endl;
    std::cout << "Model Matrix: \n" << glm::to_string(model) << std::endl;
    std::cout << "MVP Matrix: \n"   << glm::to_string(mvp) << std::endl;
    std::cout << "Local Position: " << glm::to_string(localVertex) << std::endl;
    std::cout << "Clip Position:  " << glm::to_string(clipVertex) << std::endl;
}
```

## 4. WebGPU 인터랙티브 데모로의 확장 구상

이 디버깅 개념을 웹 표준 차세대 그래픽스 API인 **WebGPU** 환경으로 확장하면 매우 흥미로운 인터랙티브 도구를 만들 수 있습니다.

### 시각화 및 인터랙티브 컨셉
WebGPU는 셰이더 언어로 **WGSL (WebGPU Shading Language)**을 사용합니다. 호스트 웹 페이지(TypeScript/JavaScript)에서 객체를 이동, 회전, 크기 조절할 때 실시간으로 계산되는 $M, V, P$ 행렬 데이터를 WebGPU Canvas 옆의 오버레이 패널에 실시간 텍스트로 출력하는 시스템입니다.

1.  **인터랙티브 컨트롤**: 사용자가 마우스 드래그로 3D 공간 상의 카메라 각도를 조절하거나 오브젝트를 회전시킵니다.
2.  **WebGPU Compute Shader**: 복잡한 행렬 곱셈 연산($P \cdot V \cdot M$)을 WebGPU 컴퓨트 셰이더에서 병렬로 수행할 수 있습니다.
3.  **Readback & 시각화**: 
    *   컴퓨트 셰이더 혹은 버텍스 셰이더에 사용되는 Uniform Buffer의 행렬 데이터를 CPU 쪽(JS/TS)으로 가져옵니다(`mapAsync` 이용).
    *   화면 좌측에는 3D로 렌더링되는 정점의 실시간 위치를 보여주고, 우측에는 GLM의 `string_cast`처럼 포맷팅된 $4 \times 4$ 행렬과 정점의 벡터 값을 인터랙티브 텍스트로 렌더링합니다.
    *   행렬의 특정 성분(예: 평행 이동 성분인 $m_{30}, m_{31}, m_{32}$)을 마우스 오버하면, 3D 뷰포트 상에서 해당 축(X, Y, Z)의 가이드라인이 강조 표시되는 인터랙션을 구현하여 수학적 개념과 실제 화면의 변화를 직관적으로 일치시킵니다.
