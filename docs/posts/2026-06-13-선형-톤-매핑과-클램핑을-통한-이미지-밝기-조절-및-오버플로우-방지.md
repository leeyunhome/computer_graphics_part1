# 선형 톤 매핑과 클램핑을 통한 이미지 밝기 조절 및 오버플로우 방지

컴퓨터 그래픽스 배움의 여정에서 첫걸음 중 하나는 이미지의 픽셀 데이터를 직접 제어해 보는 것입니다. 이번 커밋에서 학습자는 이미지 처리 및 블룸(Bloom) 효과의 전처리 단계로 활용되는 **선형 톤 매핑(Linear Tone Mapping)**의 기초를 성공적으로 구현했습니다. 

단순히 밝기 값을 곱하는 것에서 나아가, 데이터 범위를 안전하게 제한하는 **클램핑(Clamping)**을 적용하여 그래픽스 파이프라인에서 빈번히 발생하는 데이터 오버플로우 문제를 해결한 논리적 흐름이 돋보입니다.

---

## 학습자가 직접 구현하며 이해한 핵심 개념

### 1. 선형 밝기 조절 (Linear Brightness Scaling)
이미지의 밝기를 높이기 위해 각 픽셀의 RGB 채널에 스칼라 가중치($1.5$)를 곱하는 선형 연산을 적용했습니다. 이는 컴퓨터 그래픽스에서 노출(Exposure)을 조절하는 가장 직관적이고 기본적인 방식입니다. 기존 코드의 $1.2$배에서 $1.5$배로 가중치를 상향 조정하여 더 극적인 밝기 효과를 유도했습니다.

### 2. HDR 데이터 오버플로우와 클램핑 (Clamping)
LDR(Low Dynamic Range) 표현을 사용하는 일반적인 이미지 포맷(예: 8비트 채널당 0~255)에서 픽셀 값이 표현 범위를 초과하면 색상이 왜곡되거나 깨지는 현상이 발생합니다. 학습자는 C++ 표준 라이브러리의 `std::clamp`를 도입하여 모든 색상 채널의 최종 결과물이 $[0.0, 1.0]$ 범위 내에 머물도록 제한했습니다.

이 연산은 다음과 같은 수학적 장치를 통해 데이터를 안전하게 보존합니다.
* 값이 $1.0$을 초과할 경우 $1.0$(최대 밝기 흰색)으로 제한
* 값이 $0.0$ 미만일 경우 $0.0$(최소 밝기 검은색)으로 제한

---

## 핵심 수학 공식

학습자가 코드에 적용한 픽셀당 선형 톤 매핑 및 클램핑 공식은 다음과 같이 정의할 수 있습니다.

입력 픽셀의 색상 벡터를 $\mathbf{C}_{in} = (R_{in}, G_{in}, B_{in})$, 밝기 조절 인자를 $s$, 그리고 최종 출력 색상 벡터를 $\mathbf{C}_{out}$이라고 할 때, 각 채널별 연산은 다음과 같습니다.

$$ \mathbf{C}_{out} = \text{clamp}(\mathbf{C}_{in} \times s, \, 0.0, \, 1.0) $$

이를 개별 채널 단위의 수식으로 분리하면 다음과 같이 표현됩니다.

$$ R_{out} = \max(0.0, \min(R_{in} \times s, 1.0)) $$
$$ G_{out} = \max(0.0, \min(G_{in} \times s, 1.0)) $$
$$ B_{out} = \max(0.0, \min(B_{in} \times s, 1.0)) $$

이 연산을 통해 모든 출력 채널 값은 하드웨어가 안전하게 출력할 수 있는 가시 범위 내로 매핑됩니다.

---

## 핵심 알고리즘 구조 (의사코드)

학습자가 구현한 루프 기반의 이미지 픽셀 변환 과정을 추상화한 의사코드(Pseudo-code)입니다. CPU에서 전체 픽셀 버퍼를 순회하며 1차원 선형 메모리에 접근하는 방식을 보여줍니다.

```text
Function AdjustBrightnessAndClamp(Image, BrightnessFactor):
    // 이미지의 전체 픽셀 수 계산
    TotalPixels = Image.Width * Image.Height
    
    // 모든 픽셀을 순회하며 채널별 연산 수행
    For i From 0 To TotalPixels - 1:
        CurrentPixel = Image.Pixels[i]
        
        // 각 RGB 채널에 밝기 가중치를 곱하고 [0.0, 1.0] 범위로 제한
        CurrentPixel.R = Clamp(CurrentPixel.R * BrightnessFactor, 0.0, 1.0)
        CurrentPixel.G = Clamp(CurrentPixel.G * BrightnessFactor, 0.0, 1.0)
        CurrentPixel.B = Clamp(CurrentPixel.B * BrightnessFactor, 0.0, 1.0)
        
        // 변환된 픽셀 정보를 이미지 버퍼에 갱신
        Image.Pixels[i] = CurrentPixel
```

* **참고 (그래픽스 관점의 피드백):** 각 채널을 독립적으로 클램핑하면 원래 색상의 비율이 깨져 고대비 영역에서 '색상 왜곡(Color Shifting)'이 발생할 수 있습니다. (예: 주황색이 밝아지면서 노란색을 거치지 않고 흰색으로 급격히 수렴하는 현상) 추후에는 휘도(Luminance)를 기준으로 톤 매핑을 적용하는 기법으로 발전시킬 수 있습니다.

## WebGPU 인터랙티브 데모

이 픽셀 변환 로직을 웹 브라우저에서 GPU 하드웨어 가속을 통해 실시간으로 시각화한다면 **WebGPU Compute Shader**를 사용하여 다음과 같이 구현할 수 있습니다.

### 1. 작동 방식 및 시각화 시나리오
* **화면 구성:** 웹 브라우저 화면 좌측에는 실시간 조절 가능한 HTML 슬라이더(밝기 인자 $0.0 \sim 3.0$)가 위치하고, 우측에는 입력 이미지인 `colosseum.jpg`가 렌더링됩니다.
* **병렬 처리:** CPU에서 루프를 돌며 순차 처리하던 방식과 달리, WebGPU의 GPGPU 기술을 사용합니다. 이미지의 각 픽셀을 $16 \times 16$ 크기의 2D 워크그룹 스레드 타일에 할당하여 수백만 개의 픽셀을 동시에 병렬로 연산합니다.
* **실시간 인터랙션:** 사용자가 슬라이더를 움직여 밝기 인자(Brightness Factor)를 변경하면, 변경된 값이 Uniform Buffer를 통해 GPU로 즉시 전달됩니다. GPU 내부의 텍스처 메모리에서 연산이 수행되며 실시간으로 화면이 밝아지거나 어두워지는 모습을 60fps 이상의 부드러운 속도로 관찰할 수 있습니다.

### 2. WebGPU 핵심 컴퓨트 셰이더 (WGSL) 구조
```rust
struct Params {
    brightness_factor : f32,
}

@group(0) @binding(0) var inputTexture : texture_2d<f32>;
@group(0) @binding(1) var outputTexture : texture_storage_2d<rgba8unorm, write>;
@group(0) @binding(2) var<uniform> params : Params;

@compute @workgroup_size(16, 16)
fn main(@builtin(global_invocation_id) id : vec3<u32>) {
    let texSize = textureDimensions(inputTexture);
    
    // 이미지 영역을 벗어나는 스레드 예외 처리
    if (id.x >= texSize.x || id.y >= texSize.y) {
        return;
    }
    
    let coords = vec2<i32>(id.xy);
    let originalColor = textureLoad(inputTexture, coords, 0);
    
    // RGB 채널에 가중치 적용 및 클램핑 연산
    let scaledRGB = originalColor.rgb * params.brightness_factor;
    let clampedColor = clamp(scaledRGB, vec3<f32>(0.0), vec3<f32>(1.0));
    
    // 최종 알파 값은 유지한 채 결과 텍스처에 기록
    textureStore(outputTexture, coords, vec4<f32>(clampedColor, originalColor.a));
}
```

이 데모를 통해 학습자는 CPU 싱글 스레드 연산과 GPU 병렬 연산의 압도적인 성능 차이를 체감하고, 현대 그래픽스 파이프라인의 핵심인 셰이더 프로그래밍의 기초를 직관적으로 이해할 수 있게 됩니다.
